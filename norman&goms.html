Don Norman & Evaluation

Don Norman’s work in HCI emphasizes understanding the user’s interaction cycle with systems, focusing on how people plan, act, and interpret feedback. His theory is often illustrated through the execution–evaluation cycle, which divides interaction into two stages: Execution: The user forms a goal, decides on actions, and performs them. Evaluation: The user perceives the system’s response, interprets it, and evaluates if the goal is achieved.

Connection with Evaluation Techniques in HCI

Norman’s theory links directly to usability evaluation methods discussed in (Dix et al., Human–Computer Interaction, Ch. 9)

Observation and Post-Task Walkthroughs:

  • Observing users shows what actions were taken, but not always why.
  • A post-task walkthrough lets participants reflect on their actions, providing insight into goals, intentions, and alternative strategies.
  • This aligns with Norman’s evaluation stage — understanding user interpretation and thought processes.

Query Techniques (Interviews & Questionnaires):

  • Interviews and questionnaires capture user feedback about the system’s usability, helping evaluators understand users’ mental models and satisfaction.
  • They complement observations and walkthroughs, filling gaps in understanding why users acted in certain ways, linking to Norman’s theory of evaluation and interpretation.

DRUM System Example:

  • Tools like DRUM provide video annotation, tagging, and analytic metrics for usability studies.
  • They help systematically collect and analyze data on user performance and cognitive workload — supporting Norman’s approach to understanding how users interact with systems.

Example: Withdrawing Cash from an ATM

Seven Stages of Action

  • Forming the Goal: I want to withdraw cash.
  • Forming the Intention Decide: I will use my debit card at the ATM to get money.
  • Specifying the Action Plan: Insert card → enter PIN → select “Withdraw Cash” → choose amount → collect cash.
  • Executing the Action Do it: Insert card, type PIN, select options, press confirm.
  • Perceiving the State of the World Notice: Screen shows confirmation, cash comes out, card returned.
  • Interpreting the Perception Understand: Did the ATM dispense the correct amount? Did my balance update?
  • Evaluating the Outcome Check: Do I have the cash? Is the transaction successful? if yes → goal achieved,if no → maybe reattempt or call bank.

Gulf of execution: Confusion if the screen menu isn’t clear or buttons are hard to read, Gulf of evaluation: Uncertainty if the ATM doesn’t clearly indicate success (e.g., no receipt or unclear message).

GOMS

The GOMS model, proposed by Card, Moran and Newell, is a cognitive model used to describe how expert users perform routine tasks. It focuses on structured task execution and is mainly used to analyze expert, error-free performance.GOMS is an acronym for:

  • Goals:Goals represent what the user wants to achieve. In GOMS, goals act as memory points that guide the user and help recover from errors. Goals are organized hierarchically from high-level tasks to smaller subgoals.
  • Operators: Operators are the lowest-level actions required to complete a task. They may be physical actions (pressing a key, moving a mouse) or mental actions (reading a message). Operators form the basic building blocks of interaction.
  • Methods: Methods describe different ways to accomplish the same goal. For example, a window can be closed by clicking a menu option or pressing a shortcut key. Each method consists of a sequence of operators.
  • Selection Rules: When multiple methods exist, selection rules determine which one is used. The choice depends on the user, system state, and task context. These rules allow GOMS to predict user behavior rather than treating it as random.

Types of GOMS Models

The GOMS framework has several variations, each suited to different analysis goals:

  • Original GOMS (Card et al., 1983): Focused on expert user behavior. Assumes the user knows correct methods; mental preparation is included as M operators in KLM.
  • CMN-GOMS (Card, Moran, Newell): Combines task decomposition with operator-level timing. KLM can calculate predicted execution time for tasks.
  • Keystroke-Level Model (KLM): A subset of GOMS. Focuses on execution times of low-level actions without high-level goal analysis.
  • NGOMSL (Natural GOMS Language): Developed by Kieras. Structured, procedural representation for automated analysis. Supports hierarchical decomposition of goals into subgoals and methods.
  • CPM-GOMS (Cognitive Perceptual Motor): Accounts for parallel execution of cognitive, perceptual, and motor activities. Useful for modeling highly skilled expert behavior.

Design Focus: Real-World Impact

A GOMS-style analysis was used by NYNEX (US telephone company) before installing a new operator support system.The analysis showed:

  • The new system would slow down call processing
  • The system was abandoned

This saved millions of dollars.This example shows how GOMS can support design decisions before implementation.(Dix et al., Ch. 12, p. 424)
